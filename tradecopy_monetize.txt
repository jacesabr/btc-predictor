A web platform where users:
  - Create an account
  - Connect their Polymarket wallet/API keys
  - Browse available bots (starting with our BTC 5m predictor)
  - Choose to Paper Trade (virtual) or Real Trade (live USDC)
  - Set position sizing (% of balance or fixed $)
  - Follow/unfollow bots at any time
  - Track their personal P&L dashboard
  - Eventually build a "portfolio" of multiple bots

We control the signal source (our predictor fires UP/DOWN).
Users supply their own Polymarket API credentials.
We execute trades on their behalf via py-clob-client.

2. POLYMARKET API — HOW TRADING WORKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Polymarket runs on Polygon (chain ID 137).
Currency: USDC.e (Bridged USDC on Polygon).
Order book: Central Limit Order Book (CLOB).

Official Python client (maintained by Polymarket team):
  pip install py-clob-client
  GitHub: https://github.com/Polymarket/py-clob-client

Authentication flow:
  - User needs their wallet PRIVATE KEY (EOA/MetaMask)
    OR a proxy wallet (Magic/email wallet) with a separate signing key + funder address.
  - We instantiate ClobClient with their private key.
  - Call client.create_or_derive_api_creds() once → stores API key/secret/passphrase.
  - All subsequent order calls use these derived credentials (no need for raw key again).

Placing a market order (FOK = Fill or Kill, executes immediately):
  from py_clob_client.client import ClobClient
  from py_clob_client.clob_types import MarketOrderArgs, OrderType
  from py_clob_client.order_builder.constants import BUY, SELL

  client = ClobClient(
      "https://clob.polymarket.com",
      key=PRIVATE_KEY,
      chain_id=137,
      signature_type=0,  # 0 = EOA/MetaMask, 1 = Magic/email wallet
      funder=FUNDER_ADDRESS  # only needed for proxy/magic wallets
  )
  client.set_api_creds(client.create_or_derive_api_creds())

  order = client.create_market_order(MarketOrderArgs(
      token_id=TOKEN_ID,   # the YES or NO token ID from the market
      amount=25.0,         # USDC amount
      side=BUY,
      order_type=OrderType.FOK
  ))
  resp = client.post_order(order, OrderType.FOK)

Token IDs:
  - Each Polymarket market has two token IDs: one for YES (UP), one for NO (DOWN).
  - We already fetch the market slug via our PolymarketFeed (utils/polymarket.py).
  - From the slug, hit Gamma API to get conditionId, then CLOB API /markets to get token IDs.
  - The CLOB endpoint: GET https://clob.polymarket.com/markets?next_cursor=...
    or GET https://clob.polymarket.com/markets/{condition_id}

Rate limits: 10 orders/second per account.
Maker fees: 0% (limit orders that add liquidity). Taker fees: small %.
Settlement: automatic on-chain when market resolves. Winnings go back to user's wallet.

Required prior step (one-time per user):
  - User must approve the CTF Exchange and NegRisk Adapter contracts to spend their USDC.
  - This is an on-chain transaction — the user does it once in their wallet.
  - Without this, orders will fail. We can guide users through this in onboarding.
  - Reference: https://gist.github.com/poly-rodr/44313920481de58d5a3f6d1f8226bd5e

5. PAPER TRADING MECHANICS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- User sets starting balance (any amount, no real USDC required).
- On each bot signal, we fetch the LIVE Polymarket price (we already do this in utils/polymarket.py).
- We simulate the buy at that price, record it in trades table with mode='paper'.
- When the market resolves (we poll Gamma API for closed=True + winning outcome), we calculate P&L.
- P&L = (resolved_price - entry_price) / entry_price * position_size
  or more simply: if signal was correct → +((1/entry_price - 1) * stake), else → -stake

Paper balance is stored per-user in DB, updated after each resolved trade.
Users can set balance to any starting amount and reset whenever.

Position sizing options:
  - Fixed amount: e.g. $10 per trade (capped at available balance)
  - Percentage: e.g. 5% of current balance per trade

6. REAL TRADE EXECUTION FLOW (CUSTODIAL MODEL)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

WE HOLD CLIENT FUNDS. Custodial model chosen for full admin control.

Architecture:
  - We maintain ONE platform master wallet on Polygon (holds all USDC).
  - Each user has an internal ledger balance tracked in our DB.
  - We trade from OUR wallet on behalf of all users (allocating their share).
  - Users deposit USDC to our platform wallet address.
  - Users request withdrawals; we send USDC from our wallet to their address.
  - No user private keys needed at all — we control execution entirely.

Platform wallet setup:
  - Generate a dedicated Polygon wallet (private key stored in server env vars / HSM).
  - Pre-approve the Polymarket CTF Exchange contract once (our wallet, not per-user).
  - All orders submitted from this single wallet.
  - Internal DB tracks each user's allocated balance + open positions.

When a real-trade follower exists:

1. Bot signal fires: direction = UP, market_slug = btc-updown-5m-{timestamp}
2. Resolve token ID:
     GET https://gamma-api.polymarket.com/events?slug={slug}
     → extract conditionId from market[0].conditionId
     GET https://clob.polymarket.com/markets/{conditionId}
     → tokens[0] = YES token, tokens[1] = NO token
3. Determine which token: UP → YES token, DOWN → NO token
4. Aggregate order size across all active real followers:
     total_order = sum of each follower's sized amount
     (sizing per user: pct of their internal balance OR fixed $)
5. Submit ONE order from platform wallet for total_order amount
6. Record each user's proportional share of the order in trades table
7. On market resolution: distribute winnings proportionally, update each user's internal balance

Proportional P&L distribution:
  - user_pnl = (user_stake / total_stake) * total_payout
  - Wins credited, losses debited from internal balance in DB

Key security note:
  - Platform private key NEVER in code. Use environment variable or secret manager.
  - Encrypt it at rest with Fernet or AWS KMS.
  - All fund movements logged with timestamps and tx hashes for auditability.
  - Consider multi-sig wallet (Gnosis Safe) for production so no single point of failure.

6b. BEST PLATFORMS TO HOLD CLIENT FUNDS + EXECUTE TRADES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Goal: a third-party platform that (a) holds user USDC safely, (b) lets us
execute Polymarket trades programmatically, (c) lets us deposit/withdraw per user.

── OPTION 1: CIRCLE PROGRAMMABLE WALLETS ★ RECOMMENDED ──────────────────────
  Docs:  https://www.circle.com/developer/walkthroughs-tutorials/
  Why:   Circle literally makes USDC. Developer-controlled wallets = we hold
         the keys via Circle's infrastructure, not a raw private key on our server.
         REST API to create wallets, send USDC, check balances.
         Polygon USDC natively supported.
         Circle handles all key security (HSM, encryption, compliance).

  How it works:
    - We create one "developer-controlled" wallet per user via Circle API.
    - Each user gets a unique wallet address for deposits.
    - We call Circle's API to execute the USDC transfer to Polymarket (ERC-20 transfer
      to fund their Polymarket balance), then use py-clob-client to place orders.
    - Withdrawals: API call to transfer USDC from their Circle wallet to any address.
    - Admin credit/debit: API call, no on-chain tx needed from our server.

  Pricing: Free tier available. Pay-per-transaction for higher volume.
  Best for: Clean, regulated, scalable. Circle is licensed + compliant.
  Downside: Adds Circle as a dependency. Slight latency vs. direct wallet.

── OPTION 2: PRIVY (officially supported by Polymarket) ─────────────────────
  Docs:   https://docs.privy.io/recipes/polymarket-guide
  GitHub: https://github.com/Polymarket/privy-safe-builder-example

  Why: Polymarket has an OFFICIAL example repo using Privy for wallet auth +
       server-side signing. Privy creates a wallet per user (embedded wallet),
       handles all key management, and lets us sign transactions server-side.
       Users log in with email/social — no MetaMask needed.

  How it works:
    - Each user gets a Privy embedded wallet (they own it, we can sign for them
      server-side via quorum/policy settings).
    - We top up their wallet address with USDC.
    - Our server calls Privy's API to sign + submit Polymarket orders on their behalf.
    - Official Polymarket privy-safe-builder-example is a Next.js app showing exact flow.

  Pricing: Free up to 1000 MAU. Then ~$0.05/user/month.
  Best for: Best UX (email login, no crypto wallet required from user).
            Most aligned with Polymarket's own stack.
  Downside: More setup. Need to handle Privy → Safe → CLOB flow.
            Wallets are technically user-owned (slight legal difference from full custody).

── OPTION 3: TURNKEY ────────────────────────────────────────────────────────
  Docs:   https://docs.turnkey.com/
  GitHub: https://github.com/Polymarket/turnkey-safe-builder-example

  Why: Another platform with an official Polymarket example. Turnkey does
       server-side key management — we store keys in their secure enclave,
       call their API to sign transactions. Similar to Privy but more developer-focused.

  How it works:
    - Create a wallet (sub-org) per user via Turnkey API.
    - Fund it with USDC.
    - Call Turnkey signing API from our backend to sign + submit Polymarket orders.
    - Fully programmatic, no user interaction needed after initial setup.

  Pricing: Usage-based. Competitive with Privy.
  Best for: Pure backend/programmatic use. No frontend wallet UI needed.
  Downside: Less beginner-friendly to set up than Circle.

── OPTION 4: DIY PLATFORM WALLET (simplest, least safe) ─────────────────────
  One Polygon wallet we own. Raw private key in env vars.
  All user balances tracked in our DB. No third-party dependency.
  Fastest to build. Most risk (if key leaks, all funds lost).
  Only sensible for very early prototype / testing.

── RECOMMENDATION ────────────────────────────────────────────────────────────

  START: Circle Programmable Wallets
    → Best balance of safety, compliance, and ease of use.
    → Circle handles all key security. We just call their REST API.
    → USDC native support (Circle makes USDC).
    → Per-user wallet addresses make deposits clean.

  ALTERNATIVE: Privy
    → If you want users to log in with email (no crypto knowledge needed).
    → Polymarket has an official example to copy directly.
    → More of a "user-owned wallet, server-assisted signing" model.

  AVOID for now: Fireblocks, BitGo, Copper, Anchorage
    → Enterprise/institutional grade. $1000s/month minimums. Overkill.

7. ADMIN: ADD / WITHDRAW FUNDS (CUSTODIAL)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DEPOSITS (user adds money):
  - User sends USDC.e to the platform wallet address on Polygon.
  - We watch for incoming USDC transfers via Alchemy/Polygon webhook or polling.
  - On detection: credit user's internal balance in DB.
  - Show deposit address + QR code in their account page.
  - Minimum deposit amount configurable in admin panel.

WITHDRAWALS (user pulls money):
  - User requests withdrawal in UI (amount + their Polygon wallet address).
  - We validate: amount <= available balance (balance minus open position stakes).
  - Admin approves OR auto-process if below a threshold (e.g. auto-approve < $500).
  - We send USDC from platform wallet to user's address using web3.py:
      from web3 import Web3
      # ERC-20 transfer on Polygon
      usdc_contract.functions.transfer(user_address, amount_in_wei).transact()
  - Deduct from user's internal balance. Log tx hash.

ADMIN CREDIT / DEBIT (manual adjustment):
  - Admin panel: select user → adjust balance (add bonus, refund, deduct fee).
  - All adjustments logged with reason + admin user ID in an audit_log table.
  - No on-chain tx needed — purely internal ledger adjustment.

DB additions for custodial model:
  deposits    (id, user_id, amount_usdc, tx_hash, polygon_from_address, status, created_at)
  withdrawals (id, user_id, amount_usdc, tx_hash, polygon_to_address, status, approved_by, created_at)
  audit_log   (id, admin_user_id, target_user_id, action, amount, reason, created_at)
  platform_wallet (address, usdc_balance_cache, last_synced)

IMPORTANT LEGAL NOTE:
  Holding client funds in crypto is regulated in many jurisdictions.
  This may require a Money Transmitter License (MTL) in the US, or equivalent
  registration in other countries. Get legal advice before launching publicly.
  Options to reduce regulatory burden:
    - Operate in a jurisdiction with lighter-touch crypto rules.
    - Limit to users in specific regions.
    - Structure as a "prediction market access tool" rather than a financial service.
    - Keep balances small / add withdrawal limits while in early access/beta.

11. KEY RISKS & NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  - Private key security is critical. Never store plaintext. Use encryption + env-var master key.
  - Polymarket requires one-time on-chain contract approval per user (CTF Exchange allowance).
    This CANNOT be done via API — user must sign a tx in their wallet. Build a clear onboarding step.
  - FOK orders can fail if liquidity is thin. Add fallback: retry as GTC limit order slightly off mid-price.
  - BTC 5m markets have short windows (~5 min). Execution must happen within ~seconds of signal.
  - Polymarket ToS: check whether running automated bots on behalf of other users is permitted.
  - KYC/AML: Polymarket is permissionless but regulatory landscape is evolving.
    Non-custodial approach (users control own keys) is legally cleaner.
  - Market resolution can be delayed. Handle "pending resolution" state gracefully in P&L tracking.
